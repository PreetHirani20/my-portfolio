The Problem

Dynamic Decisions vs. Static Benchmarks

Most AI models are benchmarked on static text (like answering trivia or writing code). But in the real enterprise world—specifically logistics—decisions are dynamic. A freight forwarder makes decisions based on Pareto Tradeoffs (e.g., you cannot optimize for low carbon emissions without sacrificing speed). Achieving this optimal balance is complex, requiring an AI that doesn't just pick one efficient metric, but a balanced approach.

About the Project

A Custom RL Environment

OpenEnv Logistics is a strict, deterministic state machine that simulates global supply chain routing, complete with perishability of goods, freight consolidation economics, and random global disruptions (like port strikes or war). It exists to benchmark whether an LLM can successfully act as an autonomous dispatcher.

  • The Physics Engine (FastAPI): A local server maintaining the "state of the world." Tracks days passing, shipment locations, spoilage timers, and current carbon/cost expenditures.
  • The Mathematical Grader: A strict scoring algorithm taking the final state and outputting a score (0.0 to 1.0) based on delivery success, penalties, and Pareto efficiency bonuses.
  • The Inference Pipeline: Takes the current state, feeds it to an LLM, parses the chosen action, and advances the simulation.

How It Works

The RL Loop: Observation → Action → Reward

I defined a strict language for the AI and Server to communicate:

  • Observation: The server sends a JSON payload containing weather data, shipments, routes, and global alerts.
  • Action Space: The AI must reply with strictly formatted JSON choosing one of three actions: Wait (do nothing for a day), Dispatch (move cargo down optimal route), or Consolidate (wait at port to group cargo for a 40% cost discount).
  • Dynamic Modes: Includes easy, medium, and hard modes (featuring random port strikes) forcing the AI to reroute dynamically mid-transit.

Defensive Engineering

Bulletproofing the Pipeline

Surviving a 30-minute automated cloud grading gauntlet required strict validation and robust error handling:

  • FastAPI & Pydantic Validation: We cannot trust AI to return proper data. Pydantic acts as a validation layer; if an LLM returns a wrong data type, it prevents server contamination and asks the LLM for a proper answer again.
  • Data Formatting: Output text is stripped of backslashes and whitespace before it touches the system to prevent downstream crashes.
  • Network Timeouts: Standard Python requests will hang forever if the grader drops or an API lags. I added hard 15-second network timeouts to safely abort calls instead of freezing.
  • The Race Condition Trap: When booted in the cloud, the testing script runs in milliseconds while Uvicorn takes 1-2 seconds to bind. To prevent a ConnectionRefusedError at second zero, I built a startup ping loop that polls the server up to 15 times, waiting 1 second between each try, ensuring the test only starts when the server is fully awake.

The Problem

                        

                            Teacher Onboarding Was Entirely Manual

                        
Every new teacher on the platform had to fill in their profile fields manually — tagline,
                            about
                            me, teaching style, education, experience. Most teachers submitted a paragraph of
                            unstructured text or a resume PDF. An admin had to read it, extract the relevant
                            information, rewrite it for the platform's tone, and populate each field by hand. This took
                            15–20 minutes per teacher and didn't scale.

                    


                    

                        
What I Built

                        

                            Autonomous AI Pipeline via n8n

                        
I designed and deployed a fully autonomous n8n workflow that handles the entire onboarding
                            pipeline — from webhook trigger to database update — with zero manual intervention:

                        
    
                            
  • Webhook Trigger: The backend fires
                                a POST with teacherId, inputType (text or resume), and payload. n8n immediately responds
                                with 200 so the app never hangs.
    • 
                            
  • Smart Branching: An IF node routes the
                                payload — resume PDFs get downloaded and parsed via Extract From File; raw text bios go
                                straight to normalization.
    • 
                            
  • Content Validation: A guard node checks for
                                music-related keywords and minimum content length before sending to the AI — preventing
                                garbage-in, garbage-out.
    • 
                            
  • GPT-4.1 Extraction: An LLM Chain with a
                                master prompt extracts and rewrites the bio into 5 structured fields: tagline, aboutMe,
                                teachingStyle, educationalBackground, and professionalExperience.
    • 
                            
  • Callback POST: The structured JSON is sent
                                to platform's admin API via Bearer-authenticated POST. Fields update in the database
                                instantly.
    • 
                            
  • Telegram Alerts: Success and error
                                notifications are sent to the admin's Telegram for one-click review.
